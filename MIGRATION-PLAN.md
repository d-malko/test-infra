# Migration Plan: Talos/YAML → Pulumi + Flux GitOps

## Overview

Migrate p2bid infrastructure from ad-hoc YAML/Talos configs to a fully GitOps-driven setup:

| Layer | Tool | Responsibility |
|-------|------|----------------|
| Cluster bootstrap | **Pulumi** (Python) | Talos config generation, apply, bootstrap, kubeconfig |
| Secrets | **Pulumi ESC** + **SOPS/age** | Cluster PKI in ESC, workload secrets encrypted in Git |
| Workloads | **Flux v2** | All Kubernetes resources: operators, Helm releases, apps |
| Progressive delivery | **Flagger** | Canary/blue-green for stateless custom apps (future) |

---

## Security Issues to Fix (from audit)

### 🔴 Critical — fix before anything else

| Issue | Fix |
|-------|-----|
| Cluster PKI keys in Git plaintext | Move to Pulumi ESC; rotate all certs/tokens |
| `kubeconfig` / `talosconfig` in Git | Remove from repo; store in ESC; add to `.gitignore` |
| PostgreSQL `51.195.15.0:5432` public | Remove LoadBalancer; internal-only via ClusterIP |
| Kubernetes API `51.195.61.13:6443` public | Firewall to known IPs only (CI/CD, VPN, developer IPs) |
| GitLab SSH `:22` public | Keep but rate-limit; consider VPN-only or non-standard port |
| No database backups | CNPG barmanObjectStore → MinIO |

### 🟡 High — fix during migration

| Issue | Fix |
|-------|-----|
| No resource limits on DBs | Add `resources:` to all CNPG clusters |
| No NetworkPolicies | Add per-namespace policies in Flux |
| HTTP without HTTPS redirect | Add HTTPRoute redirect in Gateway API |
| Runner RBAC too permissive | Scope `secrets` verbs to specific names only |
| MinIO `Insecure = true` in runner | Enable internal TLS or use HTTPS endpoint |
| No CNPG replication | Increase `instances: 3` on non-dev clusters |

## Boundary: Pulumi vs Flux

```
Pulumi owns:
  ├── Talos machine config (generated from secrets in ESC)
  ├── Talos apply → node
  ├── Talos bootstrap → cluster
  ├── kubeconfig → stored in Pulumi ESC output
  └── Flux bootstrap (installs Flux controllers + GitRepository CR)

Flux owns (everything after):
  ├── Cilium CNI
  ├── cert-manager + Let's Encrypt issuers
  ├── CloudNativePG operator
  ├── local-path-provisioner
  ├── Gateway API resources
  ├── NetworkPolicies
  ├── PostgreSQL clusters + backups
  ├── GitLab (Helm)
  └── GitLab Runner (Helm)
```

**Rule:** If it runs on the cluster after `kubectl` works → Flux. If it's needed to make `kubectl` work → Pulumi.

## Secrets Strategy

### Cluster-level secrets → Pulumi ESC

```
ESC Environment: p2bid/prod
  ├── talos.secrets       # machine token, cluster ID, cluster secret
  ├── talos.pki           # all CA/server/client keys (NOT in Git)
  ├── talos.kubeconfig    # admin kubeconfig (output after bootstrap)
  └── talos.talosconfig   # talosconfig (output after bootstrap)
```

Pulumi reads from ESC at deploy time. Nothing sensitive touches Git.

### Workload secrets → SOPS + age

```bash
# One-time setup
age-keygen -o prod.key              # keep private key in ESC / secure vault
# public key goes in .sops.yaml

# Encrypt a secret file
sops --encrypt secrets/cloudflare-token.yaml > secrets/cloudflare-token.enc.yaml

# Flux decrypts automatically using age key stored as K8s Secret in flux-system
```

`.sops.yaml`:
```yaml
creation_rules:
  - path_regex: flux/.*\.yaml$
    age: age1<prod-public-key>
```

Encrypted files are safe to commit. Flux decrypts in-cluster using the private age key (stored as K8s Secret, not in Git).

## Migration Steps

### Phase 0 — Prepare (before touching prod)

- [ ] Rotate ALL cluster secrets (bootstrap tokens, PKI certs, encryption keys)
- [ ] Remove `kubeconfig`, `talosconfig`, `controlplane.yaml` from Git history (`git filter-repo`)
- [ ] Add to `.gitignore`: `*.kubeconfig`, `talosconfig`, `*-patched.yaml`
- [ ] Set up Pulumi ESC environment, store rotated secrets there
- [ ] Generate age key, store private key in ESC, add public key to `.sops.yaml`
- [ ] Firewall `6443` and `50000` to known IPs only

### Phase 1 — Pulumi: Talos bootstrap

```python
# pulumi/infra/talos.py

import pulumi
from pulumi_talos import machine, cluster

config = pulumi.Config()

# Secrets from ESC (never hardcoded)
secrets = machine.Secrets("cluster-secrets")

# Generate controlplane config
cp_config = machine.GetConfigurationOutput(
    cluster_name="p2bid-prod",
    machine_type="controlplane",
    cluster_endpoint="https://51.195.61.13:6443",
    machine_secrets=secrets,
    config_patches=[open("patches/controlplane.yaml").read()],
)

# Apply to node
apply = machine.ConfigurationApply("apply-controlplane",
    client_configuration=secrets.client_configuration,
    machine_configuration_input=cp_config.machine_configuration,
    node="51.195.61.13",
    apply_mode="reboot",
)

# Bootstrap cluster (one-time)
bootstrap = machine.Bootstrap("bootstrap",
    client_configuration=secrets.client_configuration,
    node="51.195.61.13",
    opts=pulumi.ResourceOptions(depends_on=[apply]),
)

# Export kubeconfig to ESC
kubeconfig = cluster.Kubeconfig("kubeconfig",
    client_configuration=secrets.client_configuration,
    endpoint="51.195.61.13",
    opts=pulumi.ResourceOptions(depends_on=[bootstrap]),
)
pulumi.export("kubeconfig", kubeconfig.kubeconfig_raw)
```

### Phase 2 — Flux bootstrap via Pulumi

```python
# pulumi/infra/flux.py

from pulumi_kubernetes.helm.v3 import Release
from pulumi_flux import FluxBootstrapGit

# Flux bootstrap — points at this repo, flux/ directory
flux = FluxBootstrapGit("flux",
    url="https://github.com/serhii-p2bid/p2bid-infra",
    branch="main",
    path="flux/clusters/prod",
)
```

After this runs, Flux takes over and syncs everything from `flux/` directory.

### Phase 3 — Flux: Infrastructure

Order matters — use Flux `dependsOn` in Kustomizations:

```
cilium → cert-manager → cnpg-operator → local-path-provisioner
                      → gateway
                      → network-policies
```

Example `flux/clusters/prod/infrastructure.yaml`:
```yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: infrastructure
  namespace: flux-system
spec:
  interval: 10m
  path: ./flux/infrastructure
  prune: true
  sourceRef:
    kind: GitRepository
    name: flux-system
  dependsOn: []
  decryption:
    provider: sops
    secretRef:
      name: sops-age
```

### Phase 4 — Flux: Applications

```
databases → gitlab → gitlab-runner
```

`flux/apps/databases/` includes:
- `dev-pg.yaml` — with resource limits + backup config
- `gitlab-pg.yaml` — with resource limits + backup config + 3 instances in prod
- `backup-secret.enc.yaml` — SOPS-encrypted MinIO/GCS credentials

`flux/apps/gitlab/` includes:
- `HelmRelease` for GitLab CE
- Fixed security issues: no public PostgreSQL LB, HTTPS redirect, correct resource limits

### Phase 5 — Security hardening in Flux

`flux/infrastructure/network-policies/`:
```yaml
# databases namespace — only allow traffic from gitlab namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-gitlab-to-db
  namespace: databases
spec:
  podSelector: {}
  policyTypes: [Ingress]
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: gitlab
      ports:
        - port: 5432
```

### Phase 6 — Flagger (future, for custom apps)

For GitLab and PostgreSQL — **do not use Flagger**. They are stateful and don't support canary traffic shifting.

Flagger makes sense when the team builds custom stateless services on this cluster:

```yaml
# When you have custom microservices:
apiVersion: flagger.app/v1beta1
kind: Canary
metadata:
  name: my-api
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-api
  progressDeadlineSeconds: 60
  analysis:
    interval: 1m
    threshold: 5
    maxWeight: 50
    stepWeight: 10
    metrics:
      - name: request-success-rate
        thresholdRange:
          min: 99
      - name: request-duration
        thresholdRange:
          max: 500
```

Install Flagger via Flux `HelmRelease` when first custom app is ready.

## What Changes

| Before | After |
|--------|-------|
| YAML files applied manually | Flux reconciles from Git automatically |
| Secrets in Git plaintext | Secrets in Pulumi ESC or SOPS-encrypted |
| Talos configs hand-crafted | Generated by Pulumi from ESC secrets |
| No resource limits on DBs | Limits defined in Flux manifests |
| PostgreSQL exposed publicly | Internal ClusterIP only |
| No NetworkPolicies | Namespace isolation via Flux |
| No DB backups | CNPG barmanObjectStore configured |
| HTTP without redirect | HTTPRoute redirect rule in Gateway |
| Kubernetes API open to internet | Firewalled to known IPs |
